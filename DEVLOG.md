- **2026-07-20** **Found and fixed a real Modifier Deck bug**, reported by the user as "looks broken" after going through the deck and reshuffling. My first pass (draw 25x rapidly, edit composition, add card types) didn't catch it because I only checked for crashes/visual glitches, not card conservation. The user sent a screenshot showing "1 left, 0 discarded" — should never happen with a 20-card deck — which pointed straight at the bug: in `drawCard()` (`lib/modifierDeck.ts`), when a reshuffle-triggering card (crit `x2` or miss `Null`) was drawn *before* the draw pile was empty, the reshuffle branch did `shuffle(nextDiscard)` — only the discard pile plus the just-drawn card — and silently dropped `rest`, the remaining un-drawn cards still in hand. Every such reshuffle permanently destroyed however many cards hadn't been drawn yet, which is exactly how a fresh 20-card deck degrades to "1 left" after a normal play session. Fixed to `shuffle([...rest, ...nextDiscard])`. Verified with an 80-draw automated soak test (card total stayed at exactly 20 throughout, including multiple forced reshuffles) and a second test interleaving manual "Shuffle now" clicks with draws — also held at 20 throughout.
  - **Lesson for future testing**: when a stateful pile/deck is involved, always assert on the *conservation invariant* (total cards in all piles combined stays constant), not just "did it crash" — that's what would have caught this the first time.

- **2026-07-20** **Verified Short Rest / Long Rest against the rulebook** (`m-ender/gloomhaven-rules`, Resting p.17 — Frosthaven's resting rules are unchanged from Gloomhaven) and corrected `ActionCards.tsx`, which had both rests wrong. The rulebook: a rest needs **2+ cards in the discard pile** and **always loses exactly one card**. **Short rest** loses one card at **random** (not chosen), with a one-time option to **suffer 1 damage to randomly lose a *different* card** instead — there is no "no loss" option, and the modifier deck is not involved at all (the old help text invented a "if you drew a −1/−2" modifier-deck mechanic that doesn't exist; that was the DEVLOG's own hypothesis, now disproven). **Long rest** loses one **chosen** discarded card, returns the rest to hand, then does **Heal 2 Self + refresh spent items**. What the app did before: short rest let you *pick* which card to lose (or none); long rest *recovered a lost card* — a completely fabricated mechanic — and lost nothing. Rewrote both: short rest now rolls a random loss into a confirm/reroll picker (reroll disappears after one use, enforcing once-per-rest), long rest is a choose-one-to-lose picker with hint text for the Heal 2 + item refresh (HP/items are tracked on other tabs, so those stay manual). Both buttons now require `used.length >= 2`. Verified the full flow in-browser: short rest random-loss → reroll switched the doomed card and vanished → confirm moved it to Lost and returned the other to hand; long rest choose-to-lose moved the picked card to Lost, returned the other to hand, and left the pre-existing lost card lost (no bogus recovery). `tsc` + `oxlint` clean.

## Notes / Decisions

- Chose **Workers** (not Pages) as the deploy target because Durable Objects (needed for real-time multiplayer state later) aren't available on Pages, and Cloudflare's native Git integration (Workers Builds) now supports Workers directly, not just Pages, so there's no tradeoff.
- Used `@cloudflare/vite-plugin` for a single-Worker full-stack setup (frontend assets + API in one deploy) rather than React Router framework mode, keeping the stack simpler for now.
- `npm create cloudflare@latest` (C3)'s interactive prompts hung non-interactively even with `--accept-defaults`/`CI=true` (variant-selection step doesn't respect flags in v2.70.12) — worked around by scaffolding with plain `npm create vite@latest` and manually wiring in `@cloudflare/vite-plugin`, `wrangler`, and `hono`.
- **Perk list, action card text/initiative, and HP-by-level are intentionally left for the user to enter**, not hardcoded from research — couldn't reliably verify Cephalofair's exact numbers/wording from available sources, and didn't want to either assert wrong game data as fact or reproduce their proprietary card text into a public repo. The user's physical components are the authoritative source anyway.
- Found and fixed a real stale-closure bug during testing: rapid-fire clicks on the deck draw button (simulated via browser automation faster than any human tap) could drop draws because `onChange(computeNext(state))` closed over a stale `state`. Fixed by switching all list/deck mutations (`ModifierDeck`, `PerkList`, `ActionCards`) to functional `setState` updates (`onChange(prev => ...)`) so each update reads the latest state regardless of dispatch timing.
